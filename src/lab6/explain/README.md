## 1. How the 8-bit Memory Address works

The CPU generates an **8-bit memory address** for load and store operations. To use the cache, this address is split into three parts: **TAG**, **INDEX**, and **OFFSET**.

```text
  7   6   5   4   3   2   1   0
┌───┬───┬───┬───┬───┬───┬───┬───┐
│   T A G   │  I N D E X  │ OFF │
│ bits[7:5] │  bits[4:2]  │[1:0]│
└───┴───┴───┴───┴───┴───┴───┴───┘
```

- **OFFSET (bits [1:0]):** 2 bits. Selects the exact byte within a 4-byte block (0, 1, 2, or 3).
- **INDEX (bits [4:2]):** 3 bits. Maps the memory address to one of the **8 cache lines** (0 to 7).
- **TAG (bits [7:5]):** 3 bits. Used to check if the data currently stored in the cache line actually belongs to the requested memory block.

### How Address Structure Affects the FSM:
1. **Cache Hit:** The CPU requests an address. The Cache looks at the **INDEX** to find the correct line. If the `Valid` bit is 1, and the cached **TAG** matches the CPU's **TAG**, it's a Hit. The FSM stays in `IDLE` (State 0), and `BUSYWAIT` remains 0.
2. **Clean Miss:** Tag mismatch or Valid=0, AND the current line's `Dirty` bit is 0. The FSM jumps from `IDLE` to `MEM_READ_START` then `MEM_READ` to fetch the new block. `BUSYWAIT` is 1 for ~22 cycles.
3. **Dirty Miss (Conflict):** Two addresses have the *same* **INDEX** but *different* **TAGs** (e.g., `0x04` and `0x24`), and the old block was modified (`Dirty=1`). The FSM must go to `WRITE_BACK` (evicting the old block), then `WRITE_BACK_DONE`, and finally `MEM_READ` to fetch the new block. `BUSYWAIT` is 1 for ~43 cycles.

## 2. Interpreting `dcache_tb` Output

When you run `vvp dcache_tb`, it prints a cycle-by-cycle monitor of the signals. Here is how to read a typical log line:

```text
t=3170 cyc=40 ST=0 | CPU R=0 W=0 A=04 WD=xx RD=xx BW=0 | MEM mr=0 mw=0 ma=xx mbw=0
```

### Signal Key:
- **`t=3170`**: Current simulation time (in picoseconds).
- **`cyc=40`**: Clock cycle number.
- **`ST=0`**: Current Cache FSM State.
  - `0` = **IDLE** (Cache hits happen here)
  - `1` = **MEM_READ** (Waiting for 20-cycle memory fetch)
  - `2` = **WRITE_BACK** (Waiting for 20-cycle memory write-back)
  - `3` = **WRITE_BACK_DONE** (1-cycle gap before fetching)
  - `4` = **MEM_READ_START** (1-cycle gap before fetching clean miss)
- **CPU Interface (`CPU ...`)**:
  - `R`, `W`: Read/Write request from CPU.
  - `A`: 8-bit Address requested by CPU (Hex).
  - `WD`, `RD`: Write Data (from CPU), Read Data (to CPU).
  - `BW`: **`BUSYWAIT`** (1 = CPU is stalled due to a miss).
- **Memory Interface (`MEM ...`)**:
  - `mr`, `mw`: mem_read, mem_write (block requests to main memory).
  - `ma`: 6-bit block address sent to memory (`{TAG, INDEX}`).
  - `mbw`: mem_busywait (1 = Main memory is busy for 20 cycles).

### Explanation to the Instructor (Test Scenarios):
- **Test 1 (Clean Miss):** Shows the cache fetching a block for the first time. You will see `ST=4` then `ST=1` for 20 cycles while `mbw=1`.
- **Test 2 (Read Hit):** You read the same address again. `ST=0` throughout, `BW=0`. Takes 0 extra stall cycles.
- **Test 3 & 4 (Write Hit):** Writing to a cached block. FSM stays at `ST=0`.
- **Test 5 (Dirty Miss):** The cache must evict the block modified in Test 3. You will see `ST=2` (WRITE_BACK) for 20 cycles, then `ST=3`, then `ST=1` (MEM_READ) for another 20 cycles.

## 3. Interpreting `cpu_cache_tb` Output

When you run `vvp cpu_cache_tb`, the program runs the full CPU through three different scenarios.

### Explanation to the Instructor (Programs):
1. **Program 1 (Spatial Locality):**
   - The CPU reads 4 consecutive bytes (`0x04`, `0x05`, `0x06`, `0x07`).
   - Because they all map to the **same 4-byte block**, only the *first* read causes a stall (Cold Miss). The next three reads are instant Hits.
   - **Instructor Point:** "This proves spatial locality works. We fetched the whole block on the first miss, saving 60 cycles on the next 3 reads."
2. **Program 2 (Write-Allocate / Temporal Locality):**
   - The CPU writes to a block, then reads back from it.
   - The first write causes a "Fetch-then-Write" (Write-Allocate). The subsequent writes and reads are all hits because the block remains in the cache.
   - **Instructor Point:** "This shows the Write-Allocate policy. Even though we missed on a write, we loaded the block so future accesses to it were instant."
3. **Program 3 (Cache Conflict):**
   - The CPU alternates accessing `0x04` and `0x24`.
   - Both share `INDEX=1` but have different `TAG`s.
   - **Instructor Point:** "Because both addresses map to Cache Line 1, they keep evicting each other. Since we are modifying the data (Write-Back), every miss is a Dirty Miss, taking ~43 cycles each. This demonstrates the worst-case scenario for a direct-mapped cache (thrashing)."

### Performance Summary Output: Is Lab 5 Better?
At the end of Program 1, the testbench prints a comparison that makes Lab 5 look slightly faster:
```text
With Cache  : total=30  stall=22  stall%=73  CPI=3.75
Lab-5 (est.): total=28  stall=20 (5x4 mem ops)  CPI=3.50
```
If your instructor points this out and asks *"Why use a cache if Lab 5 is faster?"*, here is how to answer:

- **The "Startup Cost" (Program 1):** In Lab 5, reading 4 bytes takes `4 x 5 = 20` stall cycles. In Lab 6, the first read takes **22 stall cycles** to fetch the whole 4-byte block, but the next 3 reads are **instant**. For a tiny 4-instruction program, the 22-cycle startup cost makes it look slightly slower.
- **The Real Benefit (Program 2):** For real-world programs that access data multiple times, the cache shines. If we read/write to that same block 8 times, Lab 5 takes **40 stall cycles**, while Lab 6 still only takes **22 stall cycles** (1 miss + 7 instant hits). 
- **The Conclusion:** The cache has a high initial penalty for fetching a full block. However, thanks to Spatial and Temporal Locality in real programs, the cache will serve the vast majority of future memory requests in 0 stall cycles, making it far superior to Lab 5's architecture.
